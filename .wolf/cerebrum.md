- 2026-06-25: PR 2 完成。关键设计决策：(1) 启发式 scope 提取足够好 — plan text 中的文件路径和验证提示词是高度结构化的，80% 场景下 FILE_PATH_RE + extractVerificationHints 工作正常，PR 8 ModeContract 会替换为模型结构化输出；(2) TaskPacket→TaskTracker 转换保持向后兼容 — createTaskTracker 标记 @deprecated 但保留，pre-MasterPlan 路径和现有测试不受影响；(3) updateTaskTrackerAfterTools 的 skipLegacyStepIds gate 是最小侵入方案 — 不改函数签名结构，仅增加一个 skip 参数；(4) FILE_PATH_RE 的负向后瞻 lookbehind 是 `(?<![.\d])` 而非简单的 `\b`，因为 `\b` 会在 dotted version strings 中产生 false positive。
- 2026-06-24: Flash Triage 默认从 "off" 改为 "auto"——设计意图是替代关键词分类器，默认关闭意味着两套逻辑长期并存。改为 "auto" 后短 prompt（<240 chars）自动跳过，不影响简单交互。
- 2026-06-24: 搜索后端从 SearXNG+DuckDuckGo 迁移到 Exa+Jina Reader。原因：(1) Exa 语义搜索比关键词搜索准一个量级，(2) Jina Reader 的 Markdown 提取比 regex stripHtml 质量高很多，(3) 两者都是纯 HTTP，不需要 Docker，(4) 都有免费 tier。Agent Reach 是 Python CLI 方案，不如直接集成底层 API。DeepSeek 有原生 `web_search_20250305` API 但只能通过原生 chat API 调用，Anthropic 兼容端点不可用。
- 2026-06-24: `evaluateToolPolicy` 中 gate 3-6 都检查 `tool && ...`，如果 tool descriptor 为 undefined（如 mock 测试缺 tool 参数），这些 gate 全部静默跳过。这不是 design flaw（因为正常 loop 中 tool 始终有值），但容易误导测试编写者。测试必须提供 mock tool descriptor。
- 2026-06-25: PR 5 完成。关键设计决策：(1) module-level `activePatch` 上下文 — 因为 file.ts 工具函数不接收 loop 状态，采用和 `runtime-context.ts` 同样的模式：模块级 setter/getter，loop.ts 在节点激活时调用；(2) Phase 1 用 Set-based 行差而非 Myers diff — 因为执行时不需要精确 diff（事务记录用），精确 diff 留到 PR 8 ModeContract 的 Review 模式；(3) `checkBaseHash` 检查与磁盘写之间存在 TOCTOU 窗口 — 这是 Phase 1 的已知限制，Phase 2 会改为写 temp + rename（原子操作）；(4) Windows 大小写不敏感绕过 — `isRuntimeInternal` 统一 `.toLowerCase()` 处理，因为 Windows 文件系统不区分大小写；(5) `revisePlan` 修复 — 原先 new nodes 没有 `_packet`，导致 plan revision 后 `setActivePatchContext` 无效（所有 guard 都是 `if (cur._packet)`），现在用 `buildPacketFromLine` 构造最小可行 packet；(6) `planRef.current` 而非局部 `masterPlan` — 和前几个 PR 一样，避免 TS generator CFA narrowing 到 `never`。
- 2026-06-25: PR 6 完成。关键设计决策：(1) 证据检查集成到 `evaluateCompletionGate` 而非独立 gate — review 指出如果在 loop.ts 单独加 gate，会和已有 external completion gate 产生顺序歧义且冗余。现在 `CompletionGateInput.evidenceLedger` 可选传入，在 evaluateCompletionGate 内部与 legacy verificationEvidence 检查并列；(2) 证据摄入走 `updateTaskTrackerAfterTools` — 统一入口，避免 loop.ts 直接调用 ingestVerificationResults 导致双路径；(3) Phase 1 证据账本为累积式（additive），不做新鲜度追踪 — 旧证据不失效，与 `lastTypecheck`（每轮覆盖）形成互补；(4) manual 证据类型（addManualEvidence）仅咨询，不进入 requiredEvidenceKinds — 因为 VerificationKind 没有 manual 映射，无法从 tracker 自动要求；(5) EvidenceKind 窄于 VerificationKind（4 vs 6 种）— lint→typecheck、smoke→test、unknown→null，因为证据门只需要判断"有没有类型检查/测试/构建/人工验证"四类；(6) 保留 verificationEvidence 双写 — backward compat，避免破坏 completion-gate 和 missingTaskRequirements 中的既有检查。
- 2026-06-25: PR 7 完成。关键设计决策：(1) waiver 数据结构（RippleWaiver）附加在 obligation 上而非独立映射 — 因为豁免和 obligation 是 1:1 关系，附加比外挂 map 更简单且不易漂移；(2) waiveObligation 拒绝空/空白原因 — "waiver 必须有 reason"在函数层强制执行，调用方无法绕过；(3) mergeObligations 中新 report 覆盖旧 waiver — 语义是"新的 ripple 发现意味着上下文已变，旧豁免可能不再有效"；(4) resolveObligations 保留 waiver — 已豁免且调用方未变动的 obligation 保持豁免状态；(5) getBlockingObligations 作为统一过滤入口 — 避免多处重复 `filter(o => !o.waiver)`，降低未来 waiver 逻辑变更（如过期时间）的修改面；(6) 修复了 3 处遗漏的 `pendingRippleObligations.length` 路径（autoFinishOnVerifiedWrite/processGateOverflow/strongestRippleDecision）——这些在 RippleExitGate 和 evaluateCompletionGate 修复时被遗漏，是同一类 bug；(7) Phase 1 waiveObligation 无生产调用方 — 豁免仅通过 resolveObligations 自动创建（修改 caller 文件），手动豁免工具/提示词通路延后到后续 PR。
- 2026-06-25: PR 8 完成。关键设计决策：(1) 模块级状态模式复用 — setActiveMode/getActiveMode 沿用 PR 5 patch-transaction 的模块级 setter/getter 模式，避免在 loop.ts 中全局传参；(2) enforceModeTools 中 forbiddenTools 优先于 allowedTools — 安全策略：即使工具在 allowedTools 中，forbiddenTools 中的明确禁止仍生效；(3) MCP 工具 (mcp__*) 前缀通配 — 因为 MCP 工具名是动态的（mcp__{server}__{tool}），无法预列，统一按前缀放行（forbiddenTools 仍可精确禁止特定 MCP 工具）；(4) META_TOOLS 分类 — `request_deeper_thinking` 等元工具在全部模式中放行，避免模型自我调节能力被 mode 限制；(5) mode exit criteria 在 needsExternalCompletionGate 中触发 — 修复了 planner/review/report 等无写入模式中退出条件被绕过的 bug（原逻辑：无 tracker+无写入+无错误 → 跳过 completion gate → mode exit criteria 不检查）；(6) 通用 empty-text 检查与 mode output_not_empty 去重 — 当 mode contract 已报告 output_not_empty 时，抑制 completion gate 的通用 "model final text was empty"；(7) EvidenceKind 类型安全 — ModeExitCriterion.evidenceKind 从 string 改为 EvidenceKind，消除 `as Parameters<typeof hasEvidence>[1]` 脆弱类型转换；(8) Phase 1 限制：mode transitions 未接入 MasterPlan 节点切换（shouldTransitionMode 为 stub），mode 由 AgentOptions.activeMode 静态设置。
- 2026-06-25: PR 9 完成。🎉 9-PR 单 agent 长程运行时骨架全部完成。关键设计决策：(1) JSON 数据驱动 + TypeScript 类型安全 — case 文件是纯 JSON（满足 "input.json + fixture + expected.json" PR 要求），ReplayExpected discriminated union 在编译时检查结构一致性。JSON 不嵌入函数引用，类型在 replay-harness.ts 中定义；(2) dispatchCase 用 switch 手动映射 targetFunction → 实际函数调用 —— 因为 JSON 无法携带函数引用，switch 是显式、可测的映射方式。每个 domain 的每个 targetFunction 由开发者维护映射关系，编译时发现不匹配；(3) 断言引擎用简单字符串解析（exists/equals/gt/gte/contains）—— 覆盖 80% 验证需求（field existence, equality, numeric comparison, substring），复杂断言走 domain-specific 验证逻辑。parseAssertion 对无法解析的断言静默跳过（不抛错、不虚警）；(4) 模块级状态在每 case 前重置 —— planRef.current=null + setActiveMode("coder")，避免 cases 间污染。特别是 mode-contract 的状态和 master-plan 的共享引用；(5) Fixture 系统按需创建临时目录 —— 只有 filesystem-dependent 的 cases（checkBaseHash、createPatchTransaction）才使用 temp dir，纯函数 cases 零开销；(6) Domain-specific 验证处理异构返回类型 —— classifyEpochAction 返回 string、hasUnclosedToolChain 返回 boolean、其他返回 object，validateResult 按 string/boolean/object 三分支处理，不再假设统一返回类型；(7) 9-PR 路线图现在可以转向多 agent T3R 架构 —— 协议层（TaskPacket/PatchTransaction/Evidence/ModeContract/Ripple/Replay）均已稳定，拆 agent 只是部署问题。
